## 🔍 数据分析

### 为什么关系数量较少？

OWL 本体中的关系主要分为两类：

1. **实体间关系** (Entity-to-Entity)
   - 客体是另一个实体（有 URI 和 Label）
   - 可以转换为 Neo4j 的关系
   - 数量：**35 个**

2. **实体到属性关系** (Entity-to-Literal)
   - 客体是字面量（文本、数字等）
   - 应该作为节点的属性存储
   - 数量：**约 13,000+ 个三元组**

**当前实现**：只导入了实体间关系，字面量作为属性的部分暂未处理。

### 数据特点

✅ **优点**:
- 实体数量充足（2,586 个）
- 方剂和治法数据丰富（共 1,372 个，占 53%）
- 典籍数据完整（402 个）
- 涵盖中医养生的各个方面

⚠️ **局限**:
- 实体间关系较少（35 个）
- 大量信息存储在属性中，未形成图谱结构
- 需要进一步优化提取策略

## 💡 优化建议

### 方案 A：增强关系提取（推荐）

**目标**: 从字面量中提取更多结构化信息

**方法**:
1. 分析高频属性（如"治疗方法"、"组成"、"功效"）
2. 如果属性值是实体名称，创建关系
3. 如果属性值是列表，拆分为多个关系

**预期效果**: 关系数量提升到 500-1000 个

### 方案 B：融合 medical.json 数据

**目标**: 补充临床诊疗相关的关系

**方法**:
1. 从 medical_tcm_filtered.json 提取疾病-症状关系
2. 提取疾病-治疗方法关系
3. 与现有 OWL 数据合并

**预期效果**: 
- 实体总数达到 5,000+
- 关系总数达到 3,000+

### 方案 C：保持现状

**适用场景**: 
- 本科毕设要求不高
- 重点展示系统功能而非数据规模
- 后续可以逐步完善

## 📝 技术细节

### 构建流程

```
1. 加载 OWL 文件 (RDFLib)
   ↓
2. 提取所有个体实例
   ↓
3. 根据关键词分类实体类型
   ↓
4. 提取实体间关系
   ↓
5. 映射到 Neo4j Schema
   ↓
6. 清空 Neo4j 数据库
   ↓
7. 创建约束和索引
   ↓
8. 批量导入实体
   ↓
9. 逐个导入关系
   ↓
10. 验证导入结果
```

### 关键技术

- **RDFLib**: 解析 OWL/RDF 文件
- **Neo4j Python Driver**: 连接和操作 Neo4j
- **Cypher**: Neo4j 查询语言
- **关键词分类**: 基于中文关键词的实体分类

### 性能指标

- **构建时间**: ~2 秒
- **实体导入速度**: ~1,300 个/秒
- **关系导入速度**: ~35 个/秒（逐个导入）
- **内存占用**: < 500 MB

## 📞 问题排查

### 常见问题

**Q1: 关系数量太少怎么办？**
A: 可以采用方案 B，融合 medical.json 数据，或者优化关系提取逻辑。

**Q2: 某些实体分类不准确？**
A: 可以调整 `entity_categories` 字典中的关键词，或手动修正。

**Q3: Neo4j 连接失败？**
A: 检查 Neo4j 是否运行，密码是否正确（默认为 "password"）。

**Q4: 如何查看导入的数据？**
A: 访问 http://localhost:7474 (Neo4j Browser)，执行 Cypher 查询。
